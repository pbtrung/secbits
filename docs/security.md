# Security

Threat model and guarantees for the current design. See docs/architecture.md, docs/crypto.md, and docs/data_model.md for the mechanisms referenced here.

## What this protects against

- **Network attacker**: TLS to Firebase and InstantDB, plus end to end encryption underneath, so even a full TLS compromise only exposes ciphertext and the metadata InstantDB already sees.
- **InstantDB compromise or breach**: InstantDB stores only opaque encrypted blobs plus row ids and owner/entry links (see docs/data_model.md, Entities and Links). A full database dump exposes ciphertext and who owns which rows, never plaintext.
- **Malicious or careless InstantDB staff**: same guarantee as above; InstantDB itself never holds a decryption key.
- **Cross user access**: permission rules scope every row to `auth.id` (see docs/data_model.md, Permission rules); one authenticated user cannot read another user's `entries`, `keyStore`, or `$files` rows, the last scoped by both the `entry` link and an `auth.id`-namespaced `path`.
- **Ownership hijack via update**: `!('owner' in request.modifiedFields)` on the `entries` and `keyStore` update rules stops a user reassigning their own row onto another user's account after creation, by forbidding the `owner` link from being touched in an update at all (see docs/data_model.md, Permission rules).
- **History and entry data tampering**: `$files.update` is `false`; a file can only be created fresh (new path, new commit hash) or deleted, never edited in place, and the old file is only deleted once the new one is atomically live (see docs/crypto.md, Entry Data File).
- **Wrong key or tampered blob**: AEAD authentication covers the full blob, magic, version, salt, and ciphertext, as additional data. A single bit change anywhere, or a decrypt attempt with the wrong key, fails closed before any plaintext is returned (see docs/crypto.md, AEAD Additional Data and Security Properties).

## What this does not protect against

- **Local device compromise**: `root_master_key`, `backup_master_key`, `email`, `password`, and the R2/S3 backup access keys all live in a local config file in the clear. Anyone with read access to that file, or to process memory while the app is running with the vault decrypted, has full access. This design assumes a trusted local machine; it does not defend against malware, a compromised OS account, or physical access to an unlocked device.
- **Local unencrypted backup**: the on demand local backup is the full vault, decrypted, written to disk as plain JSON. This is a bigger exposure than the config file: reading it requires no secret at all, not even `root_master_key`. It exists deliberately as an unprotected escape hatch (see docs/features.md, Backup); anywhere that file is copied to, synced, or backed up by something else becomes a full plaintext copy of the vault.
- **Compromised Firebase credentials**: an attacker who obtains the Firebase email and password can authenticate as the user and reach their InstantDB rows, but still cannot decrypt them without `root_master_key`. This is an availability and integrity risk, they can delete or overwrite rows, including planting a duplicate `keyStore` row (see docs/data_model.md, Uniqueness), not a confidentiality one. Mitigation: use a strong, unique Firebase password not reused anywhere else, since email/password is the only factor by default; enable Firebase's built in abuse protections (email enumeration protection, sign in rate limiting) and multi factor authentication where the Firebase plan supports it. If compromise is suspected, revoke the account's refresh tokens and rotate the password from the Firebase console. Encrypted entry files (their embedded history) and cloud backups are the actual recovery path for anything an attacker deletes or overwrites; permission rules cannot prevent damage from someone who authenticates as the legitimate user.
- **Compromised build or hosting**: Cloudflare Pages, its CDN, and the repository serving the frontend JS are trusted. A compromised build could ship code that exfiltrates `root_master_key` or plaintext directly; nothing in the crypto design defends against a malicious client. Mitigation: this is a supply chain problem the crypto design cannot offset, only pipeline hygiene reduces it. Require review and branch protection before merging to main, pin dependency versions with a lockfile plus audit and Dependabot style scanning, set a restrictive Content Security Policy limiting `connect-src` to only the Firebase, InstantDB, R2, and S3 endpoints actually in use, and control exactly what triggers a Cloudflare Pages deploy. None of this eliminates the risk, it only reduces how easily a compromise happens or goes unnoticed.
- **Coarse size metadata**: ciphertext length reveals an approximate plaintext size even though content stays hidden (see docs/crypto.md, Security Properties). Mitigation: pad the compressed payload up to a fixed size bucket, e.g. the next power of two, before encryption, so length reveals only a coarse bucket rather than an exact size. Not implemented; entries in a password manager are already small and fairly uniform, so the value of the metadata leaked by exact length is low, making this a deferred, optional hardening rather than a gap that needs closing now.
- **Weak or unseeded RNG**: every guarantee here assumes the WASM RNG is properly seeded before generating salts or keys (see docs/crypto.md, Entropy Precondition). An unseeded or predictable RNG breaks confidentiality regardless of the algorithms used.
- **Duplicate `keyStore` rows**: one row per user is an assumption, not an enforced invariant (see docs/data_model.md, Uniqueness). A duplicate row is currently a fatal error condition the app must detect, not a boundary actively defended by the schema. Mitigation: the ownership pinning fix (`!('owner' in request.modifiedFields)`, see docs/data_model.md, Permission rules) already closes the attacker driven path to a duplicate row; what remains is the benign case of two tabs or devices both finding no `keyStore` row on first run. Reduce, do not try to eliminate, that race with a bootstrap lock, e.g. a short random delay plus a re check before creating. If a duplicate is ever found, the recovery heuristic is to try unwrapping existing `entries.entryKey` blobs with each candidate row's UMK; whichever one succeeds is the real row, the other is an orphan safe to delete.
- **Orphaned or duplicate `$files` rows**: `entries.entryFile` is `has: 'one'`/`has: 'one'`, matching "exactly one file per entry" exactly, made safe by the atomic delete-and-link swap (see docs/crypto.md, Entry Data File) rather than an overwrite-in-place. This is the same cardinality shape that broke once for `keyStore`↔`$users` (see docs/data_model.md, Uniqueness) — not confirmed to recur here, but if the constraint doesn't hold live, the fallback is the same as `keyStore`: `has: 'many'` plus a client side check that treats finding more than one linked file as a fatal error rather than silently picking one.
- **Cloud backups sit on infrastructure this project does not control**: R2 and every configured S3 compatible destination, `s3_config` is an array, so this can be more than one provider at once, are third parties, holding ciphertext for potentially longer retention than anything in InstantDB. Confidentiality still holds as long as `backup_master_key` is never exposed, but that same ciphertext is now also subject to each provider's own retention, access controls, and breach history, none of which this design can influence. Every additional destination configured is one more party that holds a full copy of the vault's ciphertext; it does not weaken confidentiality by itself, since all destinations share the same `backup_master_key`, but it does multiply how many places would need to stay secure for that to keep holding.

## Deliberately out of scope for now

- Sharing entries between users: would need a public/private keypair per user; not implemented.
- One time share links: would need a new, unauthenticated read path and, for a real single view guarantee, a privileged component this design otherwise avoids. Deferred.
- Rate limiting or brute force protection on the Firebase login itself: relies on whatever Firebase Authentication provides by default.

## Recovery

- **Suspected `root_master_key` compromise**: rotate it (see docs/crypto.md, Key Rotation). Only `keyStore.umkBlob` needs re encrypting.
- **Suspected UMK compromise**, or after revoking a device that cached it: rotate the UMK (see docs/crypto.md, Key Rotation). Every `entryKey` is re wrapped, atomically, in one transaction.
- **Suspected `backup_master_key` compromise**, or a suspected leak at the R2/S3 provider: change `backup_master_key` in config (see docs/crypto.md, Key Rotation). This only protects future backups; past backup objects remain under the old key unless separately re encrypted and re uploaded. Since this key is independent of `root_master_key`, rotating it never requires touching the live vault.
- Neither rotation helps against an attacker who already exfiltrated `root_master_key` along with a copy of the ciphertext from before rotation; rotation protects future compromise of stored ciphertext, not past exposure of already decrypted plaintext.
