# D1 + Workers Backend Architecture

The backend is a Cloudflare Worker backed by a D1 (SQLite) database. The Worker is the sole trust boundary between the browser and the database — all authorization is enforced here. The browser never touches D1 directly.

```text
Browser (CF Pages) → Worker (auth + CRUD API) → D1 (SQLite)
```

Authentication uses **Firebase Authentication** (email/password). The browser signs in directly to Firebase and receives a Firebase ID token (RS256 JWT). That token is forwarded as a `Bearer` credential on every Worker request. The Worker verifies it against Firebase's public keys and extracts the Firebase UID (`token.sub`) as the canonical user identity. The Worker never handles passwords or issues tokens of its own.

## D1 Schema

IDs for entries are 42-character random strings over `a-zA-Z0-9` (case-sensitive), generated client-side with `crypto.getRandomValues` before each insert. User rows use the Firebase UID as the primary key — the Worker extracts it from the verified token.

D1 maximum row size: **2,000,000 bytes**. The `value` field is capped at 1,900,000 bytes, leaving ~100 KB for the remaining columns (entry_key ~192 B, two IDs ~84 B, timestamps ~60 B).

```sql
CREATE TABLE IF NOT EXISTS users (
  id              TEXT PRIMARY KEY,               -- Firebase UID (token.sub)
  email           TEXT UNIQUE NOT NULL,
  username        TEXT NOT NULL DEFAULT '',
  user_master_key BLOB,                           -- 192 bytes; NULL until first login
  created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS entries (
  id         TEXT PRIMARY KEY,                    -- 42-char random a-zA-Z0-9
  user_id    TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  entry_key  BLOB NOT NULL,                       -- ~192 bytes wrapped key
  value      BLOB NOT NULL CHECK(length(value) < 1900000),
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_entries_user_id ON entries(user_id);
```

`password_hash` is removed. Credentials live in Firebase only. The Worker never stores or sees passwords.

## Worker API

All routes require a valid Firebase ID token in the `Authorization: Bearer <token>` header. The Worker rejects requests where `token.sub` ≠ `:userId`.

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/users/:userId/profile` | Fetch username + user_master_key |
| `POST` | `/users/:userId/profile` | Save user_master_key (first login); auto-creates user row |
| `GET` | `/users/:userId/entries` | List all raw entries |
| `GET` | `/users/:userId/entries/:entryId` | Fetch a single entry |
| `POST` | `/users/:userId/entries` | Create entry |
| `PUT` | `/users/:userId/entries/:entryId` | Update entry |
| `DELETE` | `/users/:userId/entries/:entryId` | Delete entry |

`POST /auth/login` is gone — the Worker never issues tokens.

Binary fields (`entry_key`, `value`, `user_master_key`) are base64-encoded in JSON over the wire and stored as BLOB in D1.

## Auth Design

### Firebase sign-in (browser)

The browser signs in using the Firebase REST API:

```
POST https://identitytoolkit.googleapis.com/v1/accounts:signInWithPassword?key=<FIREBASE_API_KEY>
{ "email": "...", "password": "...", "returnSecureToken": true }
→ { "idToken": "<RS256 JWT, 1 h>", "refreshToken": "...", "localId": "<UID>" }
```

The `idToken` is stored in memory and sent as `Authorization: Bearer <idToken>` on every Worker request. The `refreshToken` is used to obtain a new `idToken` when the current one nears expiry (Firebase SDK handles this automatically; custom code must call the refresh endpoint before the 1 h window).

Refresh endpoint:

```
POST https://securetoken.googleapis.com/v1/token?key=<FIREBASE_API_KEY>
{ "grant_type": "refresh_token", "refresh_token": "..." }
→ { "id_token": "...", "refresh_token": "..." }
```

### Firebase ID token verification (Worker)

Firebase ID tokens are RS256 JWTs. The Worker verifies them using Firebase's published public keys without any Google SDK. Verification steps:

1. **Fetch public keys** — GET `https://www.googleapis.com/robot/v1/metadata/x509/securetoken@system.gserviceaccount.com`. Response is a JSON object mapping `kid` → PEM certificate. Cache per the `Cache-Control: max-age=...` response header (typically ~1 h); rotate keys on cache miss.

2. **Parse the token** — split on `.`, base64url-decode header and payload (no verification yet).

3. **Select the key** — look up `header.kid` in the cached key map; reject if not found.

4. **Verify the signature** — import the PEM as a `CryptoKey` (`SubtleCrypto.importKey`, `RSA-PSS` or `RSASSA-PKCS1-v1_5` per Firebase's RS256), then `SubtleCrypto.verify` against the raw JWT bytes.

5. **Validate claims** — all of the following must hold:
   - `payload.exp > Date.now() / 1000`
   - `payload.iat <= Date.now() / 1000`
   - `payload.aud === FIREBASE_PROJECT_ID`
   - `payload.iss === "https://securetoken.google.com/" + FIREBASE_PROJECT_ID`
   - `payload.sub` is a non-empty string

6. **Extract identity** — `payload.sub` is the Firebase UID. Use it as `user_id` in D1.

The Worker requires one environment variable: `FIREBASE_PROJECT_ID` (set via `wrangler secret put FIREBASE_PROJECT_ID`).

### First-login auto-provisioning

There is no explicit user registration endpoint. Users are created in Firebase via the Firebase Console (**Authentication → Users → Add user**) or the Admin SDK / REST API. The D1 `users` row is created lazily on the first authenticated request:

- `POST /users/:userId/profile` checks whether the user row exists. If not, it inserts one using `token.sub` as `id` and `token.email` as `email` before saving `user_master_key`.
- All other routes return 404 if the user row is absent (correct — the client always calls the profile endpoint before anything else).

## Config File Format

```json
{
  "username": "Your Name",
  "worker_url": "https://<worker>.<account>.workers.dev",
  "email": "user@example.com",
  "password": "your-firebase-password",
  "firebase_api_key": "<firebase-web-api-key>",
  "firebase_project_id": "<firebase-project-id>",
  "root_master_key": "<base64-encoded key, >=256 bytes when decoded>"
}
```

| Field | Required | Description |
|---|---|---|
| `username` | Yes | Display name stored in D1 |
| `worker_url` | Yes | Cloudflare Worker base URL |
| `email` | Yes | Firebase user email |
| `password` | Yes | Firebase user password (used only for Firebase sign-in; never sent to the Worker) |
| `firebase_api_key` | Yes | Firebase Web API key from Project settings |
| `firebase_project_id` | Yes | Firebase Project ID |
| `root_master_key` | Yes | Base64-encoded random key, must decode to ≥256 bytes |

## File Layout

```text
worker/
  src/
    index.js        — HTTP router, CORS headers, all route handlers
    firebase.js     — verifyFirebaseToken(token, projectId): key fetch/cache + RS256 verify
    db.js           — D1 query helpers (users + entries)
  schema.sql        — tables above
  wrangler.toml     — local config (gitignored)
  wrangler.toml.example — template with placeholder values
  package.json      — { "type": "module" }, no runtime npm deps

scripts/
  create-firebase-user.mjs — helper: calls Firebase Auth REST API to create a user
                             (optional; Firebase Console is sufficient)
```

`auth.js` is removed (PBKDF2-SHA256 hashing and HS256 JWT signing are no longer needed).

## Client-Side API (`src/api.js`)

| Function | Transport |
|----------|-----------|
| `initApi(config)` | `POST` Firebase sign-in → stores `idToken` + `refreshToken` in memory |
| `fetchUser(userId)` | `GET /users/:userId/profile` |
| `saveUserMasterKey(userId, blob)` | `POST /users/:userId/profile` |
| `fetchRawUserDocs(userId)` | `GET /users/:userId/entries` |
| `createUserEntry(userId, entry)` | `POST /users/:userId/entries` |
| `updateUserEntry(userId, entryId, entry)` | `GET` existing + `PUT /users/:userId/entries/:entryId` |
| `restoreEntryVersion(userId, entryId, hash)` | `GET` existing + `PUT /users/:userId/entries/:entryId` |
| `deleteUserEntry(userId, entryId)` | `DELETE /users/:userId/entries/:entryId` |
| `setUserMasterKey` / `getUserMasterKey` / `clearUserMasterKey` | In-memory only (unchanged) |

`initApi` calls the Firebase sign-in REST endpoint and stores the `idToken` and `refreshToken` in module-level variables. Every subsequent Worker request reads the in-memory `idToken` and sets `Authorization: Bearer <idToken>`. A token refresh helper calls the Firebase token refresh endpoint when the current token has less than 5 minutes remaining (checked before each request using the `exp` claim decoded from the JWT payload without signature verification — the Worker performs actual verification).

Binary fields are base64-encoded over HTTP. Entry IDs are generated client-side with `crypto.getRandomValues` before each `POST`.

## Deployment

```bash
# 1. Create Firebase project
# Firebase Console → Authentication → Sign-in method → enable Email/Password
# Note the Web API key and Project ID from Project settings

# 2. Create a Firebase user
# Firebase Console → Authentication → Users → Add user
# (or use the optional create-firebase-user.mjs script)

# 3. Create D1 database
cd worker && wrangler d1 create secbits-db

# 4. Apply schema
wrangler d1 execute secbits-db --file schema.sql

# 5. Set Firebase project ID
wrangler secret put FIREBASE_PROJECT_ID
# enter <firebase-project-id> when prompted

# 6. Deploy Worker
wrangler deploy

# 7. Deploy frontend
cd .. && npm run build && wrangler pages deploy dist --project-name secbits
```

`JWT_SECRET` is no longer needed. The Worker trusts Firebase's signed tokens and uses Google's public keys for verification.
